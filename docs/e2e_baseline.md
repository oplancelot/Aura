基于现有 e2e_transcribe_wav + run_e2e_batch.ps1 与文档中的延迟/切句分析，下面是一份以 e2e 测试标准驱动优化的计划。目标仍是：低延迟、低 WER、好显示，后期接翻译。

已知缺口（脚本/结果已暴露）：

1. 指标不完整：Process_Time_s 常为 0（解析与 Audio: … \| Processing: … 同行格式不匹配）；汇总里 Avg chunk duration 仍是 N/A。
2. 无基线与门禁：没有 golden 集、p50/p95、ΔWER 阈值。
3. 延迟定义偏“离线总耗时”：缺「话音结束后 → Final 出字」的 用户可感知延迟（endpoint latency）。
4. 显示质量未量化：Provisional 目前 不做 ASR，无法评估预览正确率、闪烁、半句稳定性。
5. 场景单一：LJSpeech 干净朗读 ≠ 游戏/Discord 噪声、重叠、短促对话。

───

总体策略

把 e2e 建成 三层度量 + 一套门禁 + 一条优化闭环：

准确率 (WER) ←→ 延迟 (endpoint / ASR / 预览首帧) ←→ 显示 (分段 + 预览稳定性)
↓
参数/模型 A/B → 回归 CSV → 是否合入
↓
后期：ASR 文本 → 翻译质量 (BLEU/COMET) + 译后延迟

原则：同一套 e2e 协议 必须能同时回答「更准了吗？更快了吗？更好看了吗？」。

目标：run_e2e_batch.ps1 一次跑完就能当 官方基线，结果可复现、可 diff。

0.1 修复与补齐解析

• 正确解析 Audio: Xs | Processing: Ys | ASR: Zms（避免 Process_Time 恒 0）。

0.3 黄金对照

• 固定 offline ASR（transcribe_wav / run_asr_batch）与 E2E VAD+ASR 同一批文件。
• 定义：

• Pipeline ΔWER = WER_e2e − WER_offline
• 解释：Δ 大 → 切句/flush/漏音问题；Δ≈0 且 WER 高 → 模型/语言域问题。

交付物：一页「E2E 测试协议」+ 可重复的 baseline CSV（注明 commit）。

1.2 延迟（尽可能低）

在 e2e 中补齐 可感知延迟（比单纯 total ASR ms 更贴近产品）：

┌───────────────────────────────┬────────────────────────────────────────────────────┐
│ 指标 │ 定义 │
├───────────────────────────────┼────────────────────────────────────────────────────┤
│ Endpoint latency │ 参考句结束（或 VAD 进入 silence）→ Final 文本可用 │
├───────────────────────────────┼────────────────────────────────────────────────────┤
│ ASR latency │ Final/HardCut 触发 → ASR 返回（已有 per-chunk） │
├───────────────────────────────┼────────────────────────────────────────────────────┤
│ TTFP（Time To First Preview） │ 语音 onset → 首次 Provisional 事件（显示“有反应”） │
├───────────────────────────────┼────────────────────────────────────────────────────┤
│ Final after speech-end │ silence_close + ASR（话音落后用户等待时间） │
├───────────────────────────────┼────────────────────────────────────────────────────┤
│ RTF │ 处理时间 / 音频时长（Accuracy 模式） │
└───────────────────────────────┴────────────────────────────────────────────────────┘

当前真实瓶颈文档已写明：ASR 占 ~96%；e2e 应继续验证「改模型/线程/切句」是否真降 ASR 与 endpoint。

门禁示例：

• Final 路径 p95 ASR ≤ baseline
• 中等长度句（~2–4s 音频）endpoint p50 向 README 目标 300–550ms 收敛
• 不因追求延迟导致 ΔWER 超门禁

• 低延迟 + 好显示：快 TTFP、稳定递增预览、Final 少回写。

3.3 显示路径（UI，与 batch 互补）

• e2e 批处理 不覆盖 Direct2D 33ms timer；保留 timing\_\*.csv 抽检。
• 标准：Final 显示延迟 p95；Provisional 刷新间隔与配置一致。
• 目标：显示层不成为第二瓶颈（文档已示 ~20–33ms）。

3.4 与 offline 的分工

• offline batch：锁 ASR 上限（「模型能多准」）。
• e2e batch：锁产品路径（「用户实际看到多准、多快」）。
• 优化顺序建议：先压 ΔWER（切句别伤准）→ 再压 ASR 时延 → 再调 provisional 显示。

阶段 4：显示效果专用实验（在 e2e 上扩展）

当前 Provisional 跳过 ASR（设计如此，省延迟）。为「显示最好」需 可选模式：

1. Display-eval 模式：对 Provisional 也跑 ASR（或更轻量 partial），只用于评测，不代表线上路径。
2. 指标：
   • 预览与 Final 的 prefix match rate
   • 文本单调性（是否越改越长、少回退）
   • 首次有意义预览时间
3. 与 UI 约定：[~] provisional 替换策略（整段替换 vs 增量），用录屏/日志对照。

这样 e2e 才能量化「看起来又快又稳」。

阶段 6：后期直接翻译

在 ASR e2e 稳定后再加：

1. 管线：Final 文本 → translator（本地/Gemini）→ 译文字幕。
2. 指标：翻译延迟、BLEU/COMET（有平行语料时）、术语一致性。
3. 延迟账本：endpoint_asr + translate_ms；可并行化的地方单列实验。
4. 显示：原文/译文双轨、先出原文再贴译文 等产品策略用同一 timing 协议评。

e2e 标准扩展为：WER + MT quality + E2E latency (speech-end → translated subtitle)。

一句话总结

先把 run_e2e_batch 变成带协议、门禁、可 diff 的「产品级仪表」；再用同一套仪表按顺序优化 切句（准+显）→ ASR（延迟）→ 预览显示 → 域适配 → 翻译。
否则调参只能凭感，无法证明「更低延迟」没有牺牲 WER，也无法证明「更好看」没有更碎句。

如果你希望下一步直接落地，我建议从 M1（修脚本 + 打 baseline） 或 M2（offline vs e2e 对照报告） 开始实现；你指定一个里程碑即可
